[[section-runtime-view]]
== Vista en tiempo de ejecución


ifdef::arc42help[]
[role="arc42help"]
****
.Contents
The runtime view describes concrete behavior and interactions of the system’s building blocks in form of scenarios from the following areas:

* important use cases or features: how do building blocks execute them?
* interactions at critical external interfaces: how do building blocks cooperate with users and neighboring systems?
* operation and administration: launch, start-up, stop
* error and exception scenarios

Remark: The main criterion for the choice of possible scenarios (sequences, workflows) is their *architectural relevance*. It is *not* important to describe a large number of scenarios. You should rather document a representative selection.

.Motivation
You should understand how (instances of) building blocks of your system perform their job and communicate at runtime.
You will mainly capture scenarios in your documentation to communicate your architecture to stakeholders that are less willing or able to read and understand the static models (building block view, deployment view).

.Form
There are many notations for describing scenarios, e.g.

* numbered list of steps (in natural language)
* activity diagrams or flow charts
* sequence diagrams
* BPMN or EPCs (event process chains)
* state machines
* ...


.Further Information

See https://docs.arc42.org/section-6/[Runtime View] in the arc42 documentation.

****
endif::arc42help[]


=== Ejecución de movimiento en partida

A continuación se muestra el diagrama de secuencia del procesamiento de un movimiento durante una partida a través de la aplicación web, que representa uno de los flujos principales del sistema.

El diagrama muestra:

* La captura de entrada: interacción del usuario con el tablero hexagonal
* El procesamiento en capas: comunicación entre frontend, gateway y servicio GameY
* La resolución de movimientos: aplicación del movimiento en el juego y ejecución de bot 
* La actualización de estado: actualización del tablero y turnos según resultado

[mermaid,"Sequence diagram Game Move",png]
----
sequenceDiagram
    participant U as Usuarios
    participant W as Webapp 
    participant G as Gateway
    participant GY as Servicio GameY 
    participant Bot as Bot 

    Note over U,Bot: Usuario hace un movimiento dentro de la página del juego
    U->>W: click en celda hexagonal del tablero
    W->>W: validateMove() - verificar estado
    
    Note over W: Enviar movimiento
    W->>G: POST /game/move {state, row, col, mode, bot_id, difficulty}
    
    Note over G: Validar configuración de bot
    G->>G: validar parámetros del bot y dificultad
    G->>GY: POST /v1/game/move
    
    Note over GY: Cargar estado del juego
    GY->>GY: reconstruir tablero
    GY->>GY: verificar fin de partida
    
    Note over GY: Procesar movimiento del jugador
    GY->>GY: convertir coordenadas
    GY->>GY: aplicar movimiento 
    
    alt Juego terminado
        GY-->>G: 200 OK + game_over: true, winner
        G-->>W: 200 OK + {game_over, winner, state}
        W->>W: mostrar diálogo resultado
        W->>U: renderizar modal fin de partida
    else Modo PvP
        GY-->>G: 200 OK + state actualizado
        G-->>W: 200 OK + {game_over: false, state}
        W->>W: cambiar turno a Player 2
        W->>U: renderizar tablero actualizado
    else Modo Bot
        GY->>Bot: ejecutar turno del bot
        Bot-->>GY: coordenadas del movimiento
        GY->>GY: aplicar movimiento bot
        GY->>GY: verificar fin de partida
        GY-->>G: 200 OK + state actualizado
        G-->>W: 200 OK + {game_over, state}
        W->>W: actualizar tablero con movimiento bot
        W->>U: renderizar tablero actualizado
    end
    
    Note over U: Tablero actualizado, listo para siguiente movimiento
----


=== Ejecución de ranking de jugadores (caso de éxito)

A continuación se muestra el diagrama de secuencia de la obtención y visualización del ranking de jugadores a través de la aplicación web.

El diagrama muestra:

* El flujo de navegación: acceso a la página de ranking desde la webapp
* La inicialización de datos: carga del ranking con parámetros de ordenamiento personalizados
* El procesamiento en capas: comunicación entre frontend, gateway y servicio de usuarios
* La obtención y transformación de datos: consulta a MongoDB
* La visualización de resultados: renderizado de la tabla

[mermaid,"Sequence diagram Ranking",png]
----
sequenceDiagram
    participant U as Usuarios
    participant W as Webapp 
    participant G as Gateway
    participant S as Servicio Users
    participant BD as MongoDB 

    Note over U,BD: Usuario navega a /game/ranking
    U->>W: click en "Ranking"
    W->>W: GameRanking renderiza
    
    Note over W: useEffect - Fetching datos
    W->>G: GET /game/ranking?sortBy=wins&order=desc
    
    Note over G: Construir URL y parámetros
    G->>S: GET /game/ranking?sortBy=wins&order=desc
    
    Note over S: Validar y procesar parámetros
    S->>BD: solicitar datos
    
    BD-->>S: [{username, played, wins, losses, winRate}, ...]
    
    S-->>G: 200 OK + JSON ranking[]
    G-->>W: 200 OK + JSON ranking[]
    
    W->>W: almacenar ranking obtenido
    W->>U: renderizar tabla ranking
    
    Note over U: Ranking visible
----
 

=== Registro de un usuario

En este otro diagrama observamos el proceso de registro de un nuevo usuario en el sistema, donde participan el usuario, la API REST, el middleware de validación y la base de datos.

El proceso es el siguiente:

* El nuevo usuario envía a la API por el endpoint /createuser sus datos 
(correo, username, contraseña...)
* Los datos son validados (formato del correo, nombre de usuario, formato de la contraseña...)
* Si los datos no son válidos la API devuelve un error con código HTTP 400
* Si los datos son válidos se comprueba si el usuario no está ya creado en la base de datos
* En este caso, se hashea la contraseña introducida por el usuario, y se guardan los datos en la BDD, devolviendo un código exitoso 200 y el JSON con los datos del usuario o token de sesión

[mermaid,"Sequence diagram register",png]
----
sequenceDiagram
    participant User as Usuario (Jugador)
    participant API as Users Service (Express/Node.js)
    participant Auth as Middleware/Validación
    participant DB as BDD (MongoDB)

    Note over User, API: Intento de Registro (/createuser)
    
    User->>API: POST /createuser
    activate API
    
    API->>Auth: Validación de datos (formato, longitud, contraseña, etc.)
    
    alt Datos Inválidos
        Auth-->>API: Error de Validación
        API-->>User: 400 Bad Request (Datos inválidos)
    else Datos Correctos
        Auth-->>API: Validación Exitosa
        
        API->>DB: Verificar si el usuario ya existe
        activate DB
        DB-->>API: No existe
        deactivate DB
        
        Note right of API: Hashing de contraseña (Seguridad)
        
        API->>DB: Guardar nuevo usuario
        activate DB
        DB-->>API: Usuario guardado (Success)
        deactivate DB
        
        API-->>User: 201 Created (User Object / Token)
    end
    
    deactivate API

----
