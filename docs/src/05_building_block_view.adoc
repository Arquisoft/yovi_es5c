== Vista de Bloques

=== Sistema General (Whitebox)

[mermaid, "bb-level1-yovi", png]
----
flowchart LR
  U[Stakeholder: Usuario]
  B[Stakeholder: Bot]

  subgraph Y[Yovi]
    W[Aplicación Web]
    US[Servicio de Usuarios]
    G[Servicio de Juego / Rust]
    DB[(Base de datos)]
  end

  U -->|interactúa| W
  B -->|consume API externa| W

  W -->|gestión de usuarios| US
  W -->|lógica del juego| G
  US -->|persistencia| DB
----

Motivación::
La descomposición general separa el sistema en bloques con responsabilidades claras:
la Aplicación Web como punto de entrada y coordinador, el Servicio de Usuarios para registro/autenticación y estadísticas,
el Servicio de Juego para la lógica del juego Y y estrategias, y una Base de datos para persistencia.
Esto facilita mantenibilidad, pruebas y evolución independiente del motor de juego.

Tabla 1. Bloques de construcción contenidos (Nivel 1)

[cols="1,3", options="header"]
|===
|Bloque |Descripción

|Stakeholder: Usuario
|Persona que accede a Yovi desde un navegador para registrarse y jugar partidas.

|Stakeholder: Bot
|Cliente externo que interactúa con el sistema mediante el API expuesto para bots.

|Aplicación Web
|Punto de entrada del sistema. Ofrece la UI y expone el API para bots. Orquesta llamadas al servicio de usuarios y al servicio de juego.

|Servicio de Usuarios
|Gestiona registro/autenticación y almacenamiento de información de usuarios y estadísticas/histórico.

|Servicio de Juego (Rust)
|Implementa la lógica del juego Y: validación de estado, detección de victoria y cálculo del siguiente movimiento según estrategias.

|Base de datos
|Persistencia de usuarios, partidas e información estadística/histórico.
|===


=== Nivel 2

El Nivel 2 muestra el funcionamiento interno del sistema descomponiendo los bloques más relevantes del Nivel 1.
Se priorizan los bloques con mayor complejidad y riesgo: Aplicación Web y Servicio de Juego (Rust).

==== Whitebox: Aplicación Web

[mermaid, "bb-level2-webapp", png]
----
flowchart LR
  U[Stakeholder: Usuario]
  B[Stakeholder: Bot]

  subgraph W[Aplicación Web]
    UI[UI / Vistas]
    APIEXT[API externa para bots]
    CTRL[Controladores / Orquestación]
    CL[Clientes REST internos]
  end

  U -->|interactúa| UI
  B -->|peticiones| APIEXT

  UI --> CTRL
  APIEXT --> CTRL
  CTRL --> CL
----

Motivación::
Esta descomposición separa claramente la presentación (UI) de la lógica de orquestación y del acceso a servicios.
Además, distingue el acceso humano (UI) del acceso automatizado (API externa para bots), aunque ambos comparten lógica de aplicación.

Tabla 2. Bloques de construcción contenidos (Aplicación Web)

[cols="1,3", options="header"]
|===
|Bloque |Descripción

|UI / Vistas
|Pantallas y componentes que permiten al usuario registrarse, iniciar partidas y jugar.

|API externa para bots
|Endpoints públicos para que bots consulten/gestionen información permitida y ejecuten el método `play`.

|Controladores / Orquestación
|Lógica que valida peticiones, aplica reglas de aplicación y coordina llamadas a servicios.

|Clientes REST internos
|Capa encargada de invocar al Servicio de Usuarios y al Servicio de Juego mediante REST/JSON.
|===


==== Whitebox: Servicio de Juego (Rust)

[mermaid, "bb-level2-gamey", png]
----
flowchart TB
  subgraph G[Servicio de Juego / Rust]
    API[Interfaz Web / API REST]
    N[Notación YEN]
    CORE[Core del juego]
    STRAT[Estrategias / Bots]
  end

  API --> N
  API --> CORE
  CORE --> STRAT
----

Motivación::
El Servicio de Juego se separa en módulos internos para aislar responsabilidades:
la interfaz REST recibe peticiones, el módulo de notación interpreta/serializa posiciones en YEN,
el core valida el estado y detecta victoria, y el módulo de estrategias calcula el siguiente movimiento con distintas tácticas y niveles de dificultad.

Tabla 3. Bloques de construcción contenidos (Servicio de Juego)

[cols="1,3", options="header"]
|===
|Bloque |Descripción

|Interfaz Web / API REST
|Expone operaciones del servicio para validar estados y solicitar el siguiente movimiento.

|Notación (YEN)
|Interpreta y genera representaciones de partida en formato YEN (JSON).

|Core del juego
|Implementa reglas del juego Y: acciones válidas, estado de la partida y condición de victoria.

|Estrategias / Bots
|Implementa estrategias de juego (y/o niveles de dificultad) para sugerir el siguiente movimiento.
|===


=== Nivel 3

En el Nivel 3 se detalla un bloque del Nivel 2 que es especialmente relevante: el Servicio de Juego (Rust),
ya que concentra la lógica del juego y las estrategias.

==== Whitebox: Core del juego

[mermaid, "bb-level3-core", png]
----
flowchart TB
  subgraph CORE[Core del juego]
    STATE[Estado de partida]
    RULES[Reglas / Movimientos válidos]
    WIN[Detección de victoria]
    COORD[Coordenadas / Tablero]
  end

  RULES --> STATE
  COORD --> RULES
  STATE --> WIN
----

Motivación::
Este nivel separa los conceptos principales del dominio del juego Y:
representación de tablero/coordendas, reglas de movimientos, estado de partida y detección de victoria.
Esta separación facilita pruebas unitarias y evita acoplar reglas con estrategias.

Tabla 4. Bloques de construcción contenidos (Core del juego)

[cols="1,3", options="header"]
|===
|Bloque |Descripción

|Estado de partida
|Estructuras de datos que representan la situación actual del juego (turno, piezas, tamaño, etc.).

|Reglas / Movimientos válidos
|Lógica que determina si una acción es válida y cómo afecta al estado.

|Detección de victoria
|Algoritmos para determinar si una partida ha sido ganada según las reglas del juego Y.

|Coordenadas / Tablero
|Representación de posiciones del tablero (p.ej. coordenadas baricéntricas o índices).
|===
